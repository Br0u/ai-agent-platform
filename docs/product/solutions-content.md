## 0. 结论先行

当前解决方案模块的问题不是“内容太少”，而是不同性质的内容堆在了同一层：

- “元启 AI 开发一体化、智能办公、智能导办、视觉检索”是面向具体业务问题的方案。
- “华鲲元启 AI 全栈”是跨场景的总体参考架构和转型路径。
- “TGHCI AI 超融合”是基础设施方案。
- “制造、金融、政务”等是行业入口，不应与具体方案并列成重复详情页。
- “知识库、工作流、OCR、模型训练、智能体”是能力或组件，不应被包装成尚未确认的独立产品。

因此，解决方案中心应从“六个产品介绍页”改造成一条清晰的企业采购决策链：

> 客户处于什么场景 → 正面临什么业务问题 → 华鲲如何组合平台、智能体、模型、算力和服务 → 如何接入现有系统 → 如何分阶段实施 → 用什么指标验收 → 下一步如何咨询。

建议保留现有六个页面对象和路由，重构它们的内容角色，不需要推倒前端：

| 页面对象 | 门户显示名称 | 内容角色 | 发布优先级 |
|---|---|---|---|
| yuanqi-ai-development | 企业 AI 应用开发解决方案 | 业务方案；原“元启 AI 开发一体化” | P0 |
| visual-retrieval | 视觉智能分析与主动治理解决方案 | 业务方案；原“视觉检索一体化” | P0 |
| intelligent-guidance | 政务智能导办与辅助预审解决方案 | 行业业务方案 | P0 |
| intelligent-office | 企业智能办公解决方案 | 业务方案；正式状态待确认 | P0，确认后发布 |
| yuanqi-ai-full-stack | 企业 AI 全栈建设解决方案 | 总体参考架构与转型路径 | P1 |
| tghci-ai | AI 超融合基础设施解决方案 | 基础设施与资源承载方案 | P1 |

一级导航不再扩展一批资料不足的“金融智能化、智能客服、数据分析、智能研发”等空页面。这些方向先作为标签、场景卡或内容储备，待具备真实产品组合、交付边界和案例后再升级为独立方案。

## 2. 当前内容为什么“看起来很多，重点却很浅”

### 2.1 分类维度互相打架

当前同时按“一体化、全栈、基础设施、行业、场景、产品组合”分类。客户无法判断这些分类之间是上下级、替代关系，还是不同购买入口。

修正原则：一级只区分“解决业务问题”和“提供技术底座”；行业、角色、部署方式作为筛选标签。

### 2.2 标题是方案，正文仍是产品功能清单

“知识库、低代码、模型管理、智能问答”可以解释产品能做什么，却没有回答：

- 哪类客户应当选择？
- 现有流程的阻塞点在哪里？
- 哪些组件是必选，哪些是按需？
- 需要客户准备什么数据和系统？
- 交付完成的定义是什么？

修正原则：每个能力都必须连接到业务问题、落地环节或验收结果。

### 2.3 缺少企业采购最关心的中后段内容

现有内容重点集中在 Hero、能力和场景，缺少：

- 适用客户与触发事件
- 产品组合及责任边界
- 现有系统集成
- 部署选择与前置条件
- POC 到生产的实施阶段
- 交付物、验收口径与服务边界
- 可公开案例和证据

修正原则：详情页必须支撑业务负责人、技术负责人、采购和交付团队共同决策。

### 2.4 大量数字看似“有证据”，实际不可直接发布

原资料包含并发、处理时长、准确率、路数、模型性能、客户效果等数字，但多数缺少软件版本、模型版本、硬件配置、输入规模、测试数据、网络环境、统计周期和客户授权。

修正原则：数字不是越多越企业级。没有完整测试条件的数字应进入待核实字段，而不是成为 Hero 指标。

### 2.5 页面结构过度一致也会削弱方案差异

六个页面可以共享决策框架，但不应共享完全一样的视觉叙事：

| 方案 | 应有的专属内容模块 |
|---|---|
| 企业 AI 应用开发 | 从业务需求到智能体上线的开发流水线 |
| 视觉智能分析 | 视频接入—语义解析—规则判断—人工复核—处置闭环 |
| 政务智能导办 | 群众问、传、填、审、办的服务旅程与权责边界 |
| 企业智能办公 | 部门 × 助手 × 企业知识的工作矩阵 |
| 企业 AI 全栈 | 分层参考架构与分阶段建设路线图 |
| AI 超融合 | 工作负载 × 资源池 × 运维控制面的承载模型 |

## 3. 解决方案模块定位

### 3.1 一句话定位

面向企业数字化与智能化转型场景，基于华鲲元启平台、行业智能体、模型服务、TGHCI 和鲲鹏/昇腾算力，为客户提供从业务咨询、方案设计、部署实施到持续运营的端到端 AI 解决方案。

### 3.2 核心价值主张

不是出售一个孤立的软件功能或一台服务器，而是帮助客户完成三件事：

1. 从真实业务问题中选出适合 AI 的场景。
2. 把算力、模型、知识、智能体、现有系统和服务组合成可运行方案。
3. 用明确的交付物、责任边界和验收指标，把试点转成可持续运营的生产系统。

### 3.3 与产品中心的边界

| 产品中心回答 | 解决方案中心回答 |
|---|---|
| TGDataXAI 是什么，有哪些模块 | 如何用 TGDataXAI 和其他组件解决企业 AI 应用开发难题 |
| TGHCI 有哪些虚拟化和运维能力 | 如何用 TGHCI 承载通用业务、AI 和数据平台混合负载 |
| 办公智能体有哪些功能 | 如何组合办公智能体、知识工程、模型与私有化部署提升组织效率 |
| 服务器有哪些规格 | 在特定场景、数据规模和并发目标下如何选型 |

同一内容不得在两个中心重复维护。解决方案页只写产品在当前方案中的角色，并链接到产品详情和兼容矩阵。

## 4. 信息架构

### 4.1 推荐一级结构

    解决方案 /solutions
    ├─ 业务解决方案
    │  ├─ 企业 AI 应用开发
    │  ├─ 企业智能办公
    │  ├─ 政务智能导办与辅助预审
    │  └─ 视觉智能分析与主动治理
    ├─ 技术底座
    │  ├─ 企业 AI 全栈建设
    │  └─ AI 超融合基础设施
    └─ 发现入口
       ├─ 按行业
       ├─ 按业务目标
       ├─ 按部署方式
       └─ 按建设阶段

“按行业”只负责聚合，不立即生成新的行业详情页。例如制造业入口可以同时聚合企业 AI 应用开发、视觉巡检、智能办公和全栈建设。

### 4.2 方案首页内容顺序

1. Hero：客户目标，而不是产品口号。
2. “从哪里开始”：按业务目标选择。
3. 四个业务方案：突出问题、对象、结果和成熟度。
4. 两个技术底座：解释如何承载和规模化。
5. 行业入口：只显示已有真实场景与案例数量。
6. 从 POC 到生产：统一交付方法。
7. 可信证据：已授权案例、测试报告、兼容矩阵、文档。
8. 咨询 CTA：带入所选方案和场景。

以下方向可以在 CMS 建立草稿，但不得因导航完整性而编造页面：

- 企业知识管理解决方案

它们升级为正式页面的最低条件是：已确认客户问题、真实产品组合、参考架构、部署边界、交付物、至少一个可复用场景，以及一项可公开证据。

## 5. 目标用户与采购决策链

| 角色 | 首要问题 | 页面必须提供 |
|---|---|---|
| 业务负责人 | 能解决什么问题，是否适合本部门 | 适用场景、业务流程、预期结果、案例 |
| 企业决策者 | 为什么现在做，投资如何分阶段 | 建设路径、业务价值、规模化方式 |
| CIO / IT 架构师 | 如何部署、如何连接现有系统 | 架构、部署模式、依赖、兼容矩阵 |
| AI / 开发团队 | 模型、数据、接口和扩展能力如何 | 产品角色、开发流程、文档与 API |
| 安全与合规 | 数据在哪里，谁能访问，AI 能否替代人工决定 | 数据边界、权限、审计、人工复核 |
| 采购与项目经理 | 买什么、交付什么、如何验收 | 产品组合、服务范围、里程碑、验收项 |
| 运维团队 | 上线后如何监控、升级和恢复 | 运维、SLA、备份、升级和故障边界 |

页面阅读顺序应覆盖完整决策链：

    识别问题 → 判断适配 → 理解组合 → 评估技术可行性
    → 确认交付边界 → 核验证据 → 发起方案评估

## 6. 每个方案的标准内容模型

所有方案共用决策框架，但只要求约 60% 结构一致，剩余 40% 使用专属模块表达差异。

### 6.1 必备模块

| 顺序 | 模块 | 必须回答 |
|---|---|---|
| 01 | 决策摘要 | 谁适合、解决什么、核心组合、部署方式、成熟度 |
| 02 | 客户挑战 | 当前流程为何低效或不可持续 |
| 03 | 目标与边界 | 方案要改变什么，不承担什么 |
| 04 | 问题—组合—结果映射 | 每个产品为何存在于这套方案 |
| 05 | 参考架构 | 各层关系、数据流、现有系统连接和责任边界 |
| 06 | 核心能力 | 能力如何支撑业务，不只罗列功能 |
| 07 | 专属业务流程 | 此方案独有的角色、步骤、输入与输出 |
| 08 | 部署与集成 | 私有化、内网、云/混合方式及接口 |
| 09 | 实施路径 | 调研、POC、生产、运营各阶段交付物 |
| 10 | 前置条件 | 数据、网络、设备、标准、组织与权限 |
| 11 | 验收指标 | 业务、技术、运营指标及测试口径 |
| 12 | 案例与证据 | 授权案例、报告、兼容数据、演示或文档 |
| 13 | 安全合规 | 数据、模型、权限、审计和人工复核 |
| 14 | 相关资源 | 产品、文档、案例、兼容矩阵、FAQ |
| 15 | 下一步 | 带上下文的评估、POC 或商务咨询 |

### 6.2 方案内容字段

后续 CMS 或静态数据至少应支持：

| 字段 | 类型 | 说明 |

| category | 枚举 | business / foundation |
| maturity | 枚举 | draft / preview / published / retired |

| triggerEvents | 列表 | 何时应启动方案评估 |
| pains | 结构化列表 | 问题、影响、当前方式 |
| outcomes | 结构化列表 | 预期结果，不默认写无依据数字 |
| components | 结构化列表 | 名称、角色、必选/可选、产品链接、状态 |
| architecture | 结构化内容 | 层级、数据流、系统连接 |

### 6.3 指标发布格式

任何性能或效果指标必须同时具备：

- 指标名称与单位
- 硬件型号和数量
- 软件、模型与驱动版本
- 输入数据规模和特征
- 并发、采样或统计条件
- 测试方法与样本量
- 测试日期和责任团队
- 是否为实验室、POC 或生产数据
- 客户授权状态

缺少任一关键条件时，页面只写“支持按项目目标进行容量评估与验收”，不显示具体数值。

## 7. 方案一：企业 AI 应用开发解决方案

### 7.1 决策摘要

| 项目 | 内容 |
|---|---|
| 原始名称 | 元启 AI 开发一体化解决方案 |
| 面向对象 | 已采购算力但缺少应用闭环的组织；有业务开发团队但 AI 工程能力不足的企业；需要私有化开发治理的国央企、制造、金融、教育、科研和政务客户 |
| 触发事件 | 多个部门同时提出 AI 需求；模型、知识库和工作流分散；POC 难转生产；权限和数据边界无法统一 |
| 一句话 | 以 TGDataXAI 为开发与治理核心，组合国产算力、模型服务、知识工程、智能体编排和专家服务，建立从场景验证到生产运营的企业 AI 应用开发体系。 |
| 核心结果 | 统一开发入口、沉淀可复用 AI 资产、缩短场景验证链路、建立权限和运营边界 |
| 主要部署 | 企业内网私有化；按项目评估是否支持其他方式 |
| 资料成熟度 | 已有商业方案；兼容、性能和资产格式仍需版本化资料 |

### 7.2 客户挑战

1. 算力、模型、知识库、工作流和应用分别由不同工具管理，开发和运维链路割裂。
2. 业务部门能描述需求，但难以直接参与 AI 场景验证，全部依赖少量算法或开发人员。
3. POC 中形成的 Prompt、知识库、流程和智能体缺少统一资产模型，难以跨部门复用。
4. 模型调用、知识访问、算力使用和应用发布缺少统一权限与审计。
5. 已采购智能算力，但缺少场景咨询、模型适配和应用交付能力，硬件没有形成业务闭环。

### 7.3 方案目标与边界

方案目标：

- 建立企业级 AI 应用开发和运营控制面。
- 让业务人员参与知识问答和简单智能体的验证。
- 让专业开发者能够编排复杂工作流、集成系统并进行模型工程。
- 将知识、模型、流程、智能体和权限纳入统一治理。

方案边界：

- 零代码不等于所有复杂业务都无需开发。
- 平台不能自动保证答案正确，仍需知识治理、评测、权限和人工复核。
- 具体模型、芯片、数据库和中间件的支持范围以版本化兼容矩阵为准。

## 8. 方案二：视觉智能分析与主动治理解决方案

### 8.1 决策摘要

| 项目 | 内容 |
|---|---|
| 原始名称 | 视觉检索一体化解决方案 |
| 面向对象 | 已有大量摄像头和视频平台，但新场景响应慢、人工回看压力高的公安、城管、交通、应急、园区、电力等组织 |
| 触发事件 | 传统算法无法覆盖长尾场景；突发需求无法快速上线；现有视频只能事后回看；多厂商算法和告警分散 |
| 一句话 | 利用视觉多模态大模型和自然语言规则，把现有视频资源转化为可检索、可持续布控、可复核和可处置的主动治理能力。 |
| 核心结果 | 利旧摄像头、缩短场景验证周期、统一在线/离线检索与布控、形成告警处置闭环 |
| 主要部署 | 客户机房私有化接入现有视频系统；可先做受控 POC |
| 资料成熟度 | 已发布方案；算法数量、性能路数、协议兼容和案例数字需核实 |

### 8.2 客户挑战

1. 传统小模型依赖样本采集、标注、训练和专项开发，新需求上线周期长。
2. 大量摄像头产生的视频主要用于事后回看，人工检索耗时且容易漏看。
3. 城市治理、公共安全和园区场景变化快，固定算法难覆盖突发或长尾问题。
4. 视频设备、解析、算法、告警和处置系统分散，无法形成统一运营闭环。
5. 识别结果如果直接触发执法或高风险处置，容易产生误报和责任风险。

### 8.3 方案目标与边界

方案目标：

- 使用自然语言定义检索目标和布控条件。
- 支持实时视频与历史视频的语义检索。
- 对常态和临时任务进行持续布控、结果统计与预警管理。
- 与现有摄像头、视频综合平台和处置系统协同，而非默认替换全部既有系统。

方案边界：

- 视觉模型结果用于线索发现和辅助判断，高风险告警必须人工复核。
- “零样本”不等于所有现场环境无需调试或验收。
- 摄像头协议、码流、分辨率、网络、采样策略和场景复杂度会直接影响容量与效果。

### 8.4 产品组合

| 组成 | 角色 | 选型 |

| 视觉检索/行业视频智能体 | 检索、布控、预警、任务和规则管理 | 必选 |
| 元启 TGDataXAI | 模型服务、流程编排、权限和场景资产管理 | 必选 |
| 视觉多模态大模型 | 理解人员、物体、行为和关系 | 必选 |
| 视频解析增强能力 | 接入、解码、切帧和分析前处理 | 按现场 |
| AT3500 G3 / AT9508 G3 等算力 | 承载视觉模型和视频处理 | 按容量评估 |
| TGHCI | 资源管理、虚拟化和运维 | 按部署规模 |
| 既有摄像头、视综平台、告警/工单系统 | 视频输入、结果展示与业务处置 | 客户现有系统 |

## 9. 方案三：政务智能导办与辅助预审解决方案

### 9.1 决策摘要

| 项目 | 内容 |
|---|---|
| 原始名称 | 智能导办一体化解决方案 |
| 面向对象 | 政务服务大厅、行政审批、市场监管、园区服务和公共事业窗口 |
| 触发事件 | 咨询高峰排队；政策口径不一致；材料退补频繁；线上线下断裂；夜间和节假日无服务 |
| 一句话 | 围绕“问、识、填、审、办”组织政策知识、材料识别、表单辅助和预审工作流，为群众提供连续导办，为窗口人员提供可追溯的辅助判断。 |
| 核心结果 | 统一咨询口径、减少重复录入、提前发现材料问题、连接线上与线下办理 |
| 主要部署 | 政务内/外网按系统边界部署，与现有综窗、办件系统、小程序或自助终端集成 |
| 资料成熟度 | 已发布方案；事项范围、接口、指标和案例授权需项目化确认 |

### 9.2 客户挑战

1. 政策咨询依赖窗口和热线，高峰期等待时间长，夜间与节假日覆盖不足。
2. 政策更新后，网站、移动端、热线与窗口的口径可能不同步。
3. 办事材料类型多、填写规范复杂，群众往往到现场后才发现缺失或错误。
4. 身份证、合同、地址等信息重复录入，增加群众和窗口人员负担。
5. 线上预约、材料准备、线下办理和后续审核缺少连续服务链路。

### 9.3 方案目标与责任边界

方案目标：

- 基于已审核政策和事项指南提供 7×24 小时咨询。
- 通过多轮交互识别事项与具体情形。
- 提取材料信息，形成可确认的表单草稿和问题提示。
- 在人工正式审核前完成规则化辅助预审。
- 对接现有系统，支持线上与线下双轨办理。

责任边界：

- AI 只进行导办、信息提取、草稿生成和辅助预审。
- 正式受理、行政审批、执法判断和最终决定必须由具备授权的人员与系统完成。
- 政策解释必须基于已审核、在有效期内且标明来源的知识。
- 对无法确定、政策冲突或高风险问题必须转人工。

### 9.4 产品组合

| 组成 | 角色 | 选型 |
|---|---|---|
| 智能导办智能体 | 多轮咨询、事项定位、材料提示和流程引导 | 必选 |
| 元启 TGDataXAI | 知识、流程、权限、模型和应用运营 | 必选 |
| 政策与事项知识库 | 提供版本化、可追溯的办事依据 | 必选；客户确认内容 |

### 9.5 专属服务旅程

页面核心应是一条群众与窗口协同旅程：

1. 问：群众描述要办什么，系统基于有效政策回答并识别意图。
2. 定：通过多轮问题定位到事项、情形和办理渠道。
3. 传：群众上传材料，系统识别文档类型和关键字段。
4. 填：系统生成待确认表单草稿，用户核对并修改。
5. 预审：按事项规则提示缺失、格式或一致性问题。
6. 办：进入线上签名/正式系统，或打印材料到线下窗口。
7. 人工兜底：不确定、冲突、高风险或异常情况转授权人员。

## 10. 方案四：企业智能办公解决方案

### 10.1 决策摘要

| 项目 | 内容 |
|---|---|
| 原始名称 | 智能办公一体化解决方案 |
| 面向对象 | 机关、国央企、制造、金融、教育科研等重视数据边界和办公规范的组织 |
| 触发事件 | 文档生产重复；知识查找耗时；合同和标书审核依赖个人；会议纪要与待办难沉淀；SaaS 不满足数据要求 |
| 一句话 | 组合智能写作、合同审核、投标辅助、智能会议、企业知识和元启平台，在企业数据边界内形成可治理的办公智能体套件。 |
| 核心结果 | 提升内容生产和复核效率、沉淀组织知识、贯通会议闭环、让规则和模板可持续运营 |
| 主要部署 | 企业内网私有化；具体远程访问和移动办公方式按安全策略设计 |
| 资料成熟度 | 原资料标注“202606 商业发布”；截至本文日期仍需产品负责人确认正式发布状态、版本和 SKU |

### 10.2 客户挑战

1. 公文、报告、合同、标书和会议纪要包含大量查找、摘录、改写与格式整理。
2. 模板、制度、条款规则和优秀案例分散在个人电脑或共享目录，难以复用。
3. 合同与投标审核依赖少数专家，标准难以统一，复核过程不可追溯。
4. 会前预约、会中记录、会后纪要、评审和待办分散在不同系统。
5. 公有 SaaS 的数据上传、模型调用和会议内容留存不符合部分企业安全要求。

### 10.3 方案目标与边界

方案目标：

- 以企业模板、制度、知识和规则约束内容生成。
- 在写作、合同、投标和会议四类流程中提供可复核的 AI 辅助。
- 将会议从预约、转写、纪要到评审发布形成连续流程。
- 在企业控制的数据和权限边界内运行并持续运营。

方案边界：

- AI 输出是草稿、风险提示或辅助意见，不能替代公文签发、法律审查、投标决策和会议责任人确认。
- “数据不出机房”只在完整部署拓扑、外部模型调用和运维访问均确认后成立。
- 文档格式、Office/WPS 兼容、语音语言和会议系统连接必须以版本化清单为准。

| 组成 | 角色 | 选型 |

### 10.6 四类流程内容

智能写作：

- 选择组织模板和文种。
- 从授权知识和用户素材中组织内容。
- 生成带来源提示的草稿。
- 完成人工事实核对、格式检查和签发。

合同审核：

- 导入合同并识别结构。
- 依据企业立场、条款库和审核规则提示风险。
- 输出风险位置、依据和修订建议。
- 由法务或授权人员复核。

投标辅助：

- 解析招标文件、资格项和评分项。
- 建立响应清单和内容大纲。
- 从企业授权资料中组织草稿。
- 检查遗漏、一致性和格式，由项目组最终确认。

智能会议：

- 会前预约、修改、取消和通知。
- 会中本地实时录音或导入授权录音。
- 会后生成结构化纪要、决议与待办草稿。
- 指定与会人评审后发布。

- 不同部门的数据分级、可见范围和输出审批
- 生成内容、风险提示和纪要的人工责任人

## 11. 方案五：企业 AI 全栈建设解决方案

它不是第五个与办公、导办并列的智能体产品，而是服务于以下客户的总体建设方案：

|---|---|
| 原始名称 | TGHCI AI 超融合解决方案 |
| 面向对象 | 需要统一承载通用业务、数据库、数据平台和 AI 负载的企业 IT 与数据中心团队 |
| 触发事件 | 传统架构设备多、资源孤岛、扩展复杂；CPU/GPU/NPU 分散；AI 环境部署和运维门槛高 |
| 一句话 | 在 X86 与 ARM 异构环境中融合计算、存储、网络、安全和运维，统一管理 CPU、GPU、NPU 与虚拟机/容器工作负载，为企业提供可扩展的 AI 基础设施。 |
| 核心结果 | 提升资源统一管理能力、降低多套系统运维复杂度、按节点扩展、支撑通用业务与 AI 协同 |
| 资料成熟度 | 已有方案资料；关键兼容、HA、容灾、规模上限和配置表仍需产品核验 |

### 12.2 客户挑战

- 传统服务器、集中式存储、FC/以太网络和安全设备形成多套管理面。
- 通用计算、AI 计算和数据平台资源分散，利用率和容量难统一规划。
- GPU/NPU 直通、虚拟化、驱动、框架和模型环境需要多个团队手工维护。
- 扩容和迁移涉及多个设备与供应商，业务上线周期长。
- 核心业务、数据库、AI 训练推理混部时缺少资源隔离和故障边界。

### 12.3 方案架构

    应用与工作负载
    ├─ OA / 会议 / 桌面云 / ERP
    ├─ OpenGauss / MySQL / 数据平台
    └─ AI 训练 / 推理 / 模型服务
             ↓
    TGHCI 统一管理与智能运维
    ├─ 计算虚拟化
    ├─ 存储虚拟化
    ├─ 网络与安全虚拟化
    ├─ K8S / 容器服务
    ├─ 异构算力管理与调度
    └─ 监控、告警、巡检、日志
             ↓
    鲲鹏 / X86 / GPU / NPU / 网络 / 本地与第三方存储

### 12.4 核心能力

| 能力组 | 内容 | 需进一步确认 |
|---|---|---|
| 异构算力 | CPU、GPU、NPU 统一管理，直通、vGPU/vNPU 和调度 | 支持型号、切分方式、超分和限制 |
| 计算虚拟化 | 虚拟机、迁移、克隆、快照、备份和 HA | 兼容 OS、迁移条件和 HA 策略 |
| 存储虚拟化 | 存储池、本地磁盘和第三方存储 | 副本/纠删码、容量、性能和兼容 |
| 网络虚拟化 | QoS、OVS、VLAN 和 Hash 策略 | 网络插件、带宽和冗余设计 |
| 容器与 AI | K8S、容器、框架、模型部署和任务管理 | K8S/运行时/驱动/框架版本 |
| 智能运维 | 监控、告警、亚健康、一键巡检、数字孪生和日志 | 指标、告警、日志留存和 API |
| 弹性扩展 | 以节点为步长的 Scale-out/Scale-in | 最小/最大节点、数据再均衡和退出条件 |

“代码自主率 >95%”“吞吐提升”等表述需要可审计的统计方法或测试报告后才能公开。

### 12.5 典型工作负载

| 工作负载 | 关注点 | 方案表达 |
|---|---|---|
| 通用虚拟化 | 稳定、迁移、备份与资源利用 | 统一承载 OA、会议、桌面和 ERP |
| 数据库 | 时延、IO、HA 和数据保护 | 按数据库认证与测试结果选型 |
| AI 推理 | NPU/GPU、模型、时延与弹性 | 预置或自动化部署需以版本清单为准 |
| AI 训练 | 高带宽、存储、调度与隔离 | 大规模训练需专项网络与容量设计 |
| 混合负载 | 资源争用与故障域 | 建立资源池、优先级和隔离策略 |
| 边缘 | 轻量、远程运维和断网 | 按节点和网络条件评估 |

### 12.6 实施路线

1. 资产盘点：服务器、加速卡、存储、网络、业务和依赖。
2. 容量与风险评估：资源峰值、故障域、迁移窗口和业务等级。
3. 架构设计：节点、网络、存储、资源池、租户和备份。
4. 小规模验证：关键硬件、OS、驱动、AI 框架和迁移流程。
5. 生产部署：集群、监控、告警、权限、日志和高可用。
6. 工作负载迁移：按低风险到核心业务分批迁移。
7. 运维移交：巡检、扩容、升级、备份、恢复和故障演练。

### 12.7 验收维度

- 硬件、OS、加速卡、驱动和框架兼容
- 资源发现、分配、隔离、回收和调度
- 虚拟机与容器生命周期
- 存储、网络、HA、备份和恢复
- 节点故障、网络故障和容量告警
- 扩容、迁移、升级和回滚
- 监控、日志、审计和运维权限
- 按目标工作负载进行性能和稳定性测试

### 12.8 待补资料

- 支持的服务器、CPU、GPU、NPU、OS、驱动和固件矩阵
- vGPU/vNPU 的产品范围和限制
- K8S、容器运行时、网络/存储插件和 AI 框架版本
- HA、故障域、副本、RPO/RTO、备份与容灾
- 单集群节点和资源上限、扩缩容条件
- 第三方存储、网络和安全产品兼容
- 源文档中错位的入门/标准/高性能配置表原稿

## 15. 内容发布与审核规则

### 15.1 可以公开

- 经审核的定位、客户问题、目标和责任边界
- 经过产品确认的架构、能力和产品角色
- 不含敏感数据的部署方法、实施步骤和标准交付物
- 已授权案例和带完整条件的测试报告
- 正式版本的兼容矩阵、文档和 FAQ

### 15.2 只在核实后公开

- 模型、芯片、服务器和软件版本组合
- 准确率、并发、时延、吞吐、视频路数和效果提升
- 客户名称、Logo、现场照片、评价和主管部门认可
- “业内最优、领先三代、修复 20+ Bug、性能提升 20%”等比较性陈述
- 智能办公正式发布状态
- 公司资本、资产、排名和服务中心数量

### 15.3 永不进入公开解决方案页

- 软件底价、渠道价、建议成交价、利润空间
- 销售话术、客户清单、渠道拓展策略
- 未经法务审核的竞品贬损和客户报价对比
- 内部联系人、测试账号、网络拓扑敏感信息
- 未脱敏的政务、视频、合同、会议和个人数据

### 15.4 已发现的待核实问题

| 问题 | 处理 |
|---|---|
| 视觉算法总数存在 43、49、交通 15/实际 9 等冲突 | 不展示总数，先建立版本化目录 |
| 城市治理场景中火灾识别重复 | 去重并由产品确认 |
| “YSL 文件”可能为术语或拼写问题 | 确认前使用通用表述 |
| “图型交互” | 统一修正为“图形化交互” |
| 智能办公标注 202606 商业发布 | 由产品负责人确认当前真实状态 |
| TGHCI 三档配置表列错位 | 获取原始表格后重建 |
| 多项性能和客户效果缺测试条件 | 转入证据台账，不在公开页显示 |

    Draft

      → Technical Review

### 16.2 审核责任

| 内容 | 责任人 |
|---|---|
| 定位、目标客户、业务价值 | 产品经理 / 方案负责人 |

- 页面应显示“内容最近审核日期”，不是代码最后部署日期。

## 18. 内容验收标准

方案页面进入开发前，应逐项通过：

- 客户无需了解产品名称，也能判断自己是否属于目标用户。
- 痛点描述的是业务现状，不是把产品能力反写成问题。
- 每个产品组件都有明确角色、必选/可选和责任边界。
- 架构同时说明业务流、数据流和运营流中的至少一项关键关系。
- 页面说明客户需要提供什么，华鲲负责交付什么。
- 有从评估、POC、生产到运营的阶段与交付物。
- 所有数字都能追溯测试或授权；否则不展示。
- AI 辅助与人工最终责任边界清楚。
- 案例已脱敏或取得公开授权。
- 相关产品、文档、兼容矩阵、案例和咨询入口可以关联。
- 不包含价格、渠道、内部话术和未经审核的竞品比较。
- 六个页面共享结构原则，但具有各自的专属流程和表现重点。

满足以上条件后，“解决方案”才不再是泛泛的产品宣传，而会成为企业客户从问题识别、技术评估到项目咨询的采购决策入口。
