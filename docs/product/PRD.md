## 1. 产品定义

### 1.1 产品名称

AI Agent Platform 企业级统一客户门户（Customer Portal）。

### 1.2 产品定位

为企业客户提供产品认知、文档查阅、版本信息、兼容查询、客户支持、授权入口和后续生态服务的统一门户。门户覆盖公开网站、登录后控制台、运营 CMS 和基础数据后台。

一期优先建设稳定、可部署、可扩展的全站骨架。所有依赖外部系统的能力均通过标准接口隔离，不在一期实现真实业务逻辑。

### 1.3 一期目标

1. 访客可在 30 秒内理解产品定位，并找到文档、版本、部署、试用、下载和支持入口。
2. 完成统一的门户页面、文档中心、客户控制台和 CMS 页面体系。
3. 建立自建 PostgreSQL 数据库、基础账号、会话和 RBAC。
4. 为 License、下载、OpenLab 等外部能力建立稳定接口契约和替换边界。
5. 交付可通过 Docker Compose 部署到公司 Linux 服务器的生产构建。

### 1.4 一期成功标准

- 规定页面全部存在，桌面、平板和手机视口无明显布局错误。
- 站内内容能够通过 CMS 管理并持久化到 PostgreSQL。
- 管理员可以创建账号、登录、分配角色并访问授权页面。
- 未启用模块统一显示“尚未开放”，接口返回稳定错误码，不出现假成功。
- Docker Compose 可完成首次部署、数据库迁移、健康检查、备份和版本回滚。
- 核心页面通过基本性能、安全、可访问性和浏览器兼容检查。

## 2. 用户与角色

### 2.1 目标用户

- 企业决策者：了解产品能力、版本和商业化路径。
- 企业 IT 运维/架构师：查阅部署、运维、兼容和故障处理文档。
- 企业开发工程师：查看 API 文档和开发能力。
- 企业管理员：进入客户控制台查看企业相关入口。
- 潜在客户：了解产品并找到试用与商务联系入口。
- 平台运营人员：维护门户内容、导航、版本和文档元数据。

### 2.2 一期角色

| 角色 | 核心权限 |
|---|---|
| 游客 | 浏览公开门户、文档、兼容矩阵、博客和公开市场内容 |
| 注册用户 | 登录控制台，查看个人入口和未开放模块状态 |
| 内容运营 | 管理页面配置、产品、版本、博客、案例、FAQ 和导航 |
| 超级管理员 | 用户、角色、权限、站点配置和全部 CMS 权限 |

### 2.3 一期账号规则

- 账号由超级管理员创建。
- 使用用户名或邮箱加密码登录。
- 一期不开放自助注册、邮箱验证和找回密码，避免依赖 SMTP。
- 管理员可重置密码、禁用账号和强制会话失效。
- 密码不以明文存储；会话 Cookie 必须启用 HttpOnly、Secure 和 SameSite。

### 3.1 完整实现

- 全局导航、页脚、搜索入口和响应式页面框架。
- 首页、产品中心、产品模块详情、版本管理、Release Note、Roadmap。

### 3.2 仅页面与接口占位

| 模块 | 一期交付 |
|---|---|
| License | 页面、状态、类型、Provider接口、Mock、功能开关 |
| 下载中心 | 页面、资源类型、Provider接口、Mock、功能开关 |

### 3.3 明确不做

- License 生成、加密、在线/离线激活、续期、解绑和机器指纹。
- 软件包、镜像、离线包的真实上传、存储、鉴权和下载。
- OpenLab 实名认证、人工审批、自动下发授权。
- 真实支付、账单、发票、合同和客户线索同步。
- Kubernetes、多实例部署、微服务拆分和分布式缓存。
- 原生移动端应用。

## 5. 页面需求

### 5.1 全局框架

- 顶部导航包含产品、文档、版本、兼容矩阵、Marketplace、支持和登录入口。
- 页面宽度、字体、颜色、间距和交互状态遵循统一设计令牌。
- 页脚展示产品导航、支持入口、公司信息、隐私和备案入口占位。
- 当前路由、键盘焦点、悬停、禁用和加载状态清晰可见。
- 手机端点击目标不小于 44×44 px。

### 5.2 首页

页面目标：快速解释“产品是什么、如何了解、文档在哪、后续如何接入”。

必须包含：

- Hero：产品名、企业级价值说明、文档主按钮和OpenLab占位次按钮。
- 产品核心能力叙事：Agent Studio、Knowledge Base、Workflow、Model Gateway、Agent Runtime、Observability。
- 最新稳定版本和Release Note入口。
- 文档、兼容矩阵、支持等高频路径。
- 产品真实截图占位；正式素材到位后替换。
- 私有化部署与商务联系收口。

### 5.3 产品中心

- 产品总览展示定位、模块关系和企业价值。
- 每个模块有独立详情页，包含能力、适用场景、关联文档和状态。
- 不虚构性能数字、客户数量或使用效果。

### 5.4 文档中心

- 基于 Nextra 和 MDX。
- 支持左侧目录、正文目录、代码复制、上一篇/下一篇、全文搜索入口。
- 文档内容存储在代码仓库中，CMS只管理文档元数据和导航引用。
- PDF导出、OpenAPI渲染和多产品版本切换作为后续增强，不计入一期原生能力。

### 5.5 兼容矩阵

- 支持按产品版本、CPU架构、操作系统、GPU、容器和中间件筛选。
- 数据由 CMS 维护。
- 结果必须显示支持级别、最低版本、推荐版本、备注和最后更新时间。
- 无匹配结果时给出明确空状态。

### 5.6 Marketplace

- 支持资源列表、分类、搜索、筛选和详情。
- 资源类型：Agent、Workflow、插件、Prompt模板、知识库模板。
- 一期只展示资源元数据和部署说明。
- 收藏可写入本站数据库；下载和一键克隆保持禁用。

### 5.7 客户控制台

- 登录后显示统一侧栏和账号菜单。
- 首页只展示真实可用的站内信息，不展示虚假授权、下载或用量数据。
- 外部模块显示状态说明、预计用途和返回公开页面的入口。

### 5.8 CMS

- 内容支持草稿、已发布、已下线三种状态。
- 发布和下线必须记录操作者、时间和内容版本。
- 列表支持搜索、筛选、排序和分页。
- 删除默认采用软删除；超级管理员可恢复。
- 富文本和MDX内容必须执行服务端校验与安全过滤。

## 6. 占位模块规范

### 6.1 页面状态

每个占位模块至少实现：

- 未开放状态
- 开发 Mock 状态
- 加载状态
- 接口失败状态
- 无权限状态

页面不能展示可提交但无结果的表单。按钮禁用时必须解释原因。

### 6.2 接口规则

- 环境变量默认关闭所有外部模块。
- 统一状态接口：`GET /api/v1/integrations/{module}/status`。
- 禁用操作返回 HTTP `501` 与错误码 `FEATURE_DISABLED`。
- 页面通过 Provider 接口访问模块，不直接依赖未来外部系统。
- 详细契约见 `docs/api/integration-contracts.md`。

## 7. 数据模型

### 7.1 一期核心实体

- `users`：账号、状态、密码摘要、最后登录时间。
- `roles`、`permissions`、`user_roles`：基础 RBAC。
- `sessions`：数据库会话或会话撤销记录。
- `site_settings`：站点名称、SEO、联系信息和全局配置。
- `navigation_items`：导航树。
- `products`、`product_modules`：产品和模块。
- `releases`、`release_notes`：版本与更新内容。
- `posts`、`categories`、`tags`：博客和动态。
- `cases`：客户案例元数据。
- `faqs`：帮助内容。
- `compatibility_items`：兼容矩阵。
- `marketplace_items`、`favorites`：市场展示和收藏。
- `media_assets`：CMS上传文件元数据。
- `audit_logs`：管理操作审计。

### 7.2 数据约束

- 所有业务实体使用稳定ID、创建时间、更新时间和状态字段。
- URL slug必须唯一。
- 发布内容保留操作者和发布时间。
- 软删除内容不能出现在公开查询中。
- 外部占位模块一期不创建完整业务表，只保留接口类型和必要配置。

## 8. UI与设计系统

### 8.1 视觉方向

- 主色范围：蓝、靛、紫。
- 气质：企业可信、技术精确、产品导向、克制未来感。
- 不使用大面积强烈蓝紫渐变、AI光球、虚假数据图表和装饰性三栏卡片堆叠。
- 缺少Logo和产品截图时使用明确文字占位。

### 8.1.1 真实资产占位决策

一期开发不等待真实品牌与业务资产。凡依赖用户、公司或外部系统提供的内容，统一先使用可识别、可替换的占位组件：

- Logo：使用产品文字标识和`[Logo SVG待替换]`占位。
- 产品截图：使用固定比例的`[产品真实截图待替换]`占位，不绘制虚假产品界面。
- 产品架构图：使用`[正式架构图待替换]`占位；模块关系仅作为结构示意，不声明真实运行链路。
- 客户案例与客户Logo：使用空状态或`[客户案例待提供]`，不虚构客户名称。
- 公司名称、联系方式、备案和隐私信息：使用明确字段占位，不展示虚假号码或备案号。
- Release Note、兼容矩阵、博客、文档和Marketplace内容：允许使用标记为“演示数据”的最小种子数据，只用于验证页面结构。

所有占位组件必须集中管理，保留稳定尺寸和替换入口；正式资产到位后不得通过逐页手工修改替换。

### 8.2 设计令牌

最终令牌在 UI 方向确认后固定。方向稿基线：

- Primary Blue：`#3157D5`
- Structural Indigo：`#4F46E5`
- Accent Violet：`#7C3AED`
- Ink：`#111827`
- Canvas：`#F5F7FC`
- Surface：`#FFFFFF`

### 8.3 响应式基线

- Desktop：1440 px设计基准。
- Tablet：768–1024 px。
- Mobile：375–430 px。
- 信息层级和功能不因响应式布局丢失。

## 9. 技术与部署

### 9.1 技术基线

- Next.js App Router、TypeScript、Tailwind CSS、shadcn/ui。
- Nextra用于文档中心。
- PostgreSQL自建数据库。
- 数据库迁移、种子数据和服务端数据访问层。
- 单仓库模块化单体，不在一期拆微服务。

### 9.2 Docker拓扑

```text
proxy  -> web  -> db
                  ^
               backup
```

- `proxy`负责80/443、TLS、限流和安全响应头。
- `web`使用Next.js standalone生产镜像。
- `db`使用独立持久化卷，不对公网开放。
- `backup`执行定时备份，备份文件复制到异机位置。

### 9.3 环境

- 本地开发环境。
- 公司测试环境。
- 公司生产环境。
- 同一镜像通过运行时环境变量晋级，不能把生产密钥写入镜像。

## 10. 非功能需求

### 10.1 性能

- 公开首页在公司目标网络下LCP不高于2.5秒。
- API列表查询P95不高于500毫秒，不含外部占位模块。
- 页面避免不必要的大型客户端JavaScript。
- 图片使用明确尺寸、压缩和延迟加载。

### 10.2 安全

- 全站HTTPS。
- 密码安全摘要、服务端输入校验、权限校验和速率限制。
- 管理后台操作写入审计日志。
- 密钥只通过环境变量或公司密钥系统注入。
- 防止SQL注入、XSS、CSRF、开放重定向和越权访问。
- CMS上传限制文件类型、大小和访问路径。

### 10.3 可用性与恢复

- 提供 `/api/health/live` 和 `/api/health/ready`。
- 数据库每日备份，生产上线前完成恢复演练。
- 部署保留上一版本镜像和数据库迁移回退说明。
- 容器异常退出后自动重启。

### 10.4 可访问性与兼容

- 目标达到WCAG 2.1 AA基本要求。
- 键盘可操作，焦点可见，表单有标签和错误提示。
- 支持当前主流Chrome、Edge、Firefox和Safari。

## 11. 验收标准

### 11.1 页面

- 路由表中的页面均可访问。
- 导航、面包屑、上一页/下一页和返回路径正确。
- 桌面、平板、手机无横向溢出和内容遮挡。
- 所有按钮具备悬停、焦点、禁用和加载状态。

### 11.2 权限

- 游客无法访问控制台和CMS。
- 内容运营不能管理用户和角色。
- 超级管理员可以创建账号、禁用账号和分配角色。
- 服务端再次验证权限，不能只隐藏前端按钮。

### 11.3 CMS与数据

- 内容创建、编辑、发布、下线和恢复可完成。
- 公开页面只读取已发布内容。
- 数据库迁移可以在空库上完整执行。
- 种子数据可创建首个超级管理员和基础站点配置。

### 11.4 占位模块

- 功能开关关闭时显示统一未开放状态。
- 禁用接口返回`501 FEATURE_DISABLED`。
- 页面不存在假成功、假License、假下载记录或假业务数据。
- 切换到Mock Provider后可以验证页面状态，不影响生产默认配置。

### 11.5 部署

- Docker Compose可在目标服务器一次启动全部服务。
- HTTPS、数据库迁移、健康检查和备份任务正常。
- 服务器重启后服务自动恢复。
- 上一版本镜像可回滚。
